# Windows

---

## Storia

### Prima e seconda generazione

**MS-DOS** (1981): mono-utente, command line, non multiprogrammato, ispirato a CP/M.

| Anno | Versione | Note |
|---|---|---|
| 1981 | MS-DOS 1.0 (8 KB) | PC IBM 8088 (16 bit) |
| 1986 | MS-DOS 3.0 (36 KB) | PC IBM/AT (i286, max 16 MB) |
| 1990–93 | Windows 3.x | GUI sopra MS-DOS; copia dell'interfaccia Macintosh |
| 1995 | **Windows 95** | Primo S/O multiprogrammato; ma mono-utente; FS FAT; semaforo globale sul nucleo; ¼ spazio condiviso R/W tra processi |
| 1998 | Windows 98 | Modeste migliorie |
| 2000 | Windows ME | Ancora basato su MS-DOS |

**GUI — paradigma WIMP:** Windows (finestre), Icons (icone), Menus (menu), Pointing (puntamento). Può essere:
- Programma in spazio utente (GNU/Linux con X Window)
- Parte del S/O (Windows)

### Terza generazione: Progetto NT

Abbandono della base MS-DOS. Scritto in **C e C++** (max portabilità, ~16–29 M linee di codice).

| Anno | Versione | Note |
|---|---|---|
| 1993 | **NT 3.1** | Architettura microkernel; fiasco commerciale |
| 1996 | **NT 4.0** | Architettura monolitica; reintroduce interfaccia Win95 |
| 1999 | **Windows 2000** (NT 5.0) | Spazio privato per processo; plug-and-play; >29 M LOC |
| 2001 | **Windows XP** | Successo commerciale; non progettato come server |
| 2007 | **Windows Vista** | Flop (lento/pesante); più sicuro; Windows Aero |
| 2009 | **Windows 7** | Risposta a Vista; multitouch; opzionale emulatore XP |
| 2012 | **Windows 8** | Tiles; Windows Store; tablet/smartphone; efficienza energetica |

**NT 3.1 (microkernel):** servizi in processi utente; alta portabilità ma basse prestazioni.  
**NT 4.0 (monolitico):** servizi riportati nel nucleo; prestazioni migliori.

## Interfaccia di programmazione

**Win32 API:** interfaccia pubblica controllata da Microsoft; le system call interne non sono pubblicate.

- Alcune procedure includono chiamate di sistema
- Altre svolgono servizi interamente in modo utente
- Nessun rigore gerarchico

## Registry

Database di configurazione del sistema (simile a un FS), salvato in file speciali detti **hive**:

- Directory → **key**
- File → **entry** = `{nome, tipo, dati}`
- 6 chiavi principali con prefisso `HKEY_`:
  - `HKEY_LOCAL_MACHINE` — hardware, software installato, avvio
  - `HKEY_CURRENT_USER` — preferenze utente corrente
  - `HKEY_USERS`, `HKEY_CLASSES_ROOT`, `HKEY_CURRENT_CONFIG`, `HKEY_PERFORMANCE_DATA`

## Gestione dei processi

### Gerarchia delle entità

| Entità | Descrizione |
|---|---|
| **Job** | Insieme di processi gestiti come unità con limiti di risorse |
| **Processo** | Possessore di risorse; ID unico; 4 GB (2 GB utente + 2 GB nucleo); non ha stato di avanzamento |
| **Thread** | Flusso di controllo gestito dal nucleo; esegue per conto del processo; 2 stack (utente + nucleo) |
| **Fiber** | Suddivisione di thread ignota al nucleo; gestita dal sottosistema Win32 |

### Sincronizzazione tra thread

| Meccanismo | Descrizione |
|---|---|
| **Mutex** (semaforo binario) | Mutua esclusione |
| **Semaforo contatore** | Conteggio risorse |
| **Sezione critica** | Solo nello spazio del thread (non kernel object) |
| **Evento manual-reset** | Rilascio manuale; sveglia tutti i thread in attesa |
| **Evento auto-reset** | Al verificarsi, sveglia uno solo dei thread in attesa |

### Comunicazione tra thread

| Meccanismo | Descrizione |
|---|---|
| **Pipe** | Bidirezionale; byte stream o messaggi |
| **Mailslot** | Unidirezionale; funziona anche in rete |
| **Socket** | Comunicazione remota |
| **RPC** | Remote Procedure Call: invoca procedure in altri processi |
| **Memoria condivisa** | Porzioni di file mappati in memoria |

## Scheduling

### Struttura delle priorità

- **32 livelli** (0–31): 31–16 = sistema/real-time, 15–0 = ordinari
- **6 classi per processo:** Realtime, High, Above-normal, Normal, Below-normal, Idle
- **7 classi per thread:** Time-critical, Highest, Above-normal, Normal, Below-normal, Lowest, Idle
- Una **coda per livello**; thread non distinti per processo di appartenenza

**Selezione:** ricerca dalla coda a priorità massima, round robin all'interno.

### Meccanismo di scheduling

Le decisioni di scheduling sono effettuate in modo nucleo come **DPC** (*Deferred Procedure Call*):
- Thread si blocca su semaforo/I/O → già in modo nucleo
- Thread segnala un oggetto → già in modo nucleo
- Esaurimento quanto → interrupt clock → DPC per concludere

### Priorità dinamica

| Evento | Effetto sulla priorità |
|---|---|
| Completamento I/O | Aumenta (+ ampliamento temporaneo del quanto) |
| Acquisizione semaforo / evento | Aumenta |
| Quanto consumato | Decresce |
| Thread pronto non selezionato troppo a lungo | Boost temporaneo per 2 quanti (anti-starvation) |

Il quanto è più ampio per la finestra in **primo piano**.

## Gestione della memoria

### Spazio di indirizzamento

4 GB per processo (32 bit), suddivisi in 2 zone da 2 GB:

```
Alto
┌──────────────────────────────────┐
│ Zona di protezione (64 KB)       │
├──────────────────────────────────┤
│ Codice e dati del S/O           │  2 GB
│ (condivisi tra tutti i processi) │
│ Tabella delle pagine del processo│
├──────────────────────────────────┤
│ Zona di protezione (64 KB)       │
├──────────────────────────────────┤
│ Codice e dati privati            │  2 GB
│ (modo utente)                    │
├──────────────────────────────────┤
│ Zona di protezione (64 KB)       │
└──────────────────────────────────┘
Basso
```

### Traduzione degli indirizzi (2 livelli)

```
Indirizzo virtuale (32 bit):
┌──────────┬──────────┬────────────┐
│ PDE      │ PTE      │ Offset     │
│ (10 bit) │ (10 bit) │ (12 bit)   │
└──────────┴──────────┴────────────┘
      ↓          ↓
Page Directory  Page Table
```

La traduzione avviene preferibilmente via **TLB**.

### Stati delle pagine virtuali

| Stato | Descrizione |
|---|---|
| **Libera** (*free*) | Non riferita da alcun PTE; inizialmente tutte le pagine sono libere (paging-on-demand) |
| **Assegnata** (*committed*) | In uso per codice o dati; caricata da disco su page fault |
| **Prenotata** (*reserved*) | Non in uso ma non libera; facilita l'assegnazione contigua (es. 1 MB riservato per lo stack) |

### DLL e file mappati in memoria

- **DLL** (*Dynamic Link Library*): codice condiviso in sola lettura; dati statici copiati per processo (copy-on-write)
- Gli indirizzi nel codice DLL sono **relativi** (risolti dal compilatore) perché la stessa DLL può mappare a indirizzi diversi per processi diversi

### Working Set

Per ogni processo i: $\text{Min}_i \leq \#\{I_i\} \leq \text{Max}_i$

**Daemon:**
- **Balance set manager** (1 s): verifica disponibilità pagine libere
- **Working set manager:** esamina WS dei processi per rilasciarne pagine (priorizza processi con WS vicino al massimo e uso recente scarso)
- **Swapper thread:** porta in standby/modified le pagine dei processi inattivi

### Stato delle page frame

| Lista | Simbolo | Descrizione |
|---|---|---|
| **In attesa** (*standby*) | A | Rimossa dal WS, non modificata; riassegnabile senza problemi |
| **Da copiare** (*modified*) | B | Rimossa dal WS, modificata; va salvata su disco prima del riuso |
| **Libera** (*free*) | C | Non più associata a nessun processo |
| **Azzerata** (*zeroed*) | D | Contenuto obliterato; necessaria per non rivelare dati privati |
| **Difettosa** (*bad*) | E | Settore fisico difettoso; non più utilizzabile |

```
WS₁…WSₙ
  │ rilascio
  ▼
B (modified) → salva su disco → A (standby) → C (free) → D (zeroed)
  ↑
richiamo nel WS (page fault)
```

### Page frame database

Tabella indicizzata per numero di frame fisico: presenza, contatore riferimenti, WS di appartenenza, lista di appartenenza, ecc.

## NTFS

### Confronto FS Windows

| FS | Bit indirizzo | Limite |
|---|---|---|
| FAT-16 | 16 | 2 GB |
| FAT-32 | 28 | 2 TB |
| **NTFS** | 64 | Virtualmente illimitato |

### Caratteristiche generali

- Nomi fino a **255 caratteri Unicode** (2–4 B/car.); path max 32.767 car.
- Separatore di path: `\`; case-sensitive ma Win32 API lo ignora per compatibilità
- File come **aggregato di attributi** (*attribute streams*): sequenza breve (metadati) + sequenza lunga (dati, fino a 2⁶⁴ B)
- Supporto **hard link** e **symbolic link**

### Volumi e cluster

- Un volume logico = sequenza lineare di cluster (512 B – 4 KB, max teorico 64 KB)
- Un volume può coprire più partizioni / dischi
- **Cluster piccolo** → meno frammentazione interna; **cluster grande** → meno accessi a disco

### MFT — Master File Table

- Una per volume; fisicamente un **file** → può risiedere ovunque nel volume
- Sequenza lineare di ≤ 2⁴⁸ record (1–4 KB ciascuno)
- Ogni record identifica un file tramite un **indice a 48 bit** + **16 bit** di sequenza (per rilevare riusi)

**Primi 16 record riservati (metadata):**

| Record | Contenuto |
|---|---|
| 0 | Descrive l'MFT stesso |
| 1 | Mirror: copia dei primi 16 record in fondo al volume |
| 3 | Informazioni sul volume (nome, versione FS, data creazione) |
| 4 | Definizione degli attributi usati nel volume |
| altri | Radice FS, bitmap blocchi liberi, boot code, ecc. |

### Struttura dei record

Ogni record = numero variabile di coppie `<descrittore-attributo, valore>`:
- **13 attributi di sistema** predefiniti + eventuali aggiuntivi
- **Attributo residente:** il valore sta interamente nel record MFT
- **Attributo non residente:** il record contiene un puntatore a 48 bit + 16 bit di controllo (→ 64 bit totali)

**Immediate file:** file con dati < 1 KB → dati interamente nel record MFT.

### Run list (sequenze di blocchi contigui)

NTFS tenta di assegnare **blocchi contigui** (*run*). Ogni run è descritto da:
- Indirizzo del 1° blocco della sequenza
- Ampiezza della sequenza (in blocchi)

```
Record base (file con 3 run):
┌─────────────────────────────────────────────┐
│ Header │ StandardInfo │ FileName │ Data attr │
│                                │ run 1: [off|len] │
│                                │ run 2: [off|len] │
│                                │ run 3: [off|len] │
└─────────────────────────────────────────────┘
```

**Efficienza:** il numero di record dipende dalla contiguità, non dalla dimensione del file:
- 20 GB in 20 run da 1 M blocchi → 21×2×8 B = **336 B** di metadati
- 64 KB in 64 run da 1 blocco → 65×2×8 B = **1040 B** di metadati

### Record con estensioni (file grandi/frammentati)

Il record base punta a N ≥ 1 **record di estensione** che descrivono le run successive.

Se nemmeno i record di estensione bastano → la run list viene posta in un file dedicato come **attributo non residente**.

### File sparsi (*sparse files*)

File con zone interne non utilizzate (es. poste a 0 e riservate per uso futuro): NTFS li gestisce in modo speciale, non allocando fisicamente i blocchi corrispondenti alle zone azzerata, ma registrando solo i run effettivamente scritti. Il file ha una dimensione logica elevata ma occupa fisicamente solo i dati reali.

### Directory

Entry di directory = parte fissa (indice MFT del file, lunghezza nome, flag) + parte variabile (nome del file).
