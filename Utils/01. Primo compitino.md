## 1. Sistema Operativo — Concetti Base

### Definizione e ruolo

Un **Sistema Operativo** è un insieme di utilità che:
1. Offre un'**astrazione** più semplice e potente della macchina fisica (macchina virtuale)
2. **Gestisce ottimamente le risorse** fisiche e logiche (CPU, memoria, I/O)

### Architetture del S/O

| Architettura | Descrizione | Pro | Contro |
|---|---|---|---|
| **Monolitica** | Un unico file oggetto; ogni procedura può chiamarne un'altra | Semplice, prestazioni elevate | Nessun information hiding |
| **A strati** | Livelli gerarchici; ogni livello usa solo quello sotto | Modularità | Chiamate più costose |
| **Microkernel** | Nucleo minimo (processi + IPC); resto in processi utente | Flessibilità, affidabilità | Prestazioni scadenti |
| **Macchina virtuale** | Virtualizzazione hardware sotto vari S/O | Isolamento completo | Complessità |

### System call

- I servizi del S/O sono invocati tramite **procedure di libreria** che nascondono le system call
- La prima istruzione di ogni system call è una **trap**: forza il passaggio in modo privilegiato
- **Interruzioni:** asincrone (evento esterno). **System call:** sincrone (richiesta esplicita del processo)

**Sequenza di esecuzione:**
1. Il programma chiama la procedura di libreria
2. Parametri sullo stack; ID della chiamata in un registro noto al S/O
3. `trap` → modo privilegiato
4. Il S/O identifica ed esegue la chiamata; ritorna al chiamante

### Interruzioni I/O

1. Il driver programma il controllore scrivendo nei suoi registri
2. Il controllore esegue l'operazione sul dispositivo fisico
3. Il controllore delle interruzioni notifica la CPU (pin)
4. La CPU legge l'identità del dispositivo e attiva il gestore appropriato (vettore delle interruzioni)
5. PC e PSW salvati → modo privilegiato → parte immediata eseguita → parte differita schedulata

## 2. Processi e Thread

### Definizione di processo

Un **processo** è un programma in esecuzione. Ha uno spazio di indirizzamento suddiviso in:

```
↑ alto
┌────────────────┐
│   Stack        │  pila dei contesti; cresce verso il basso; R/W
├────────────────┤
│   BSS          │  dati non inizializzati; dimensione variabile; R/W
├────────────────┤
│   Data segment │  dati inizializzati (malloc); dimensione variabile
├────────────────┤
│   Text segment │  codice macchina; dimensione fissa; R (condivisibile)
└────────────────┘
↓ basso
```

### PCB — Process Control Block

| Campo | Contenuto |
|---|---|
| Identificatore (PID) | Identificatore unico del processo |
| Contesto di esecuzione | Registri CPU da ripristinare dopo sospensione |
| Stato | Pronto / Esecuzione / Attesa |
| Priorità | Iniziale e corrente |
| Diritti di accesso | Privilegi e risorse autorizzate |
| Discendenza familiare | Puntatori a genitore e figli |
| Risorse assegnate | Lista delle risorse in uso |

### Diagramma degli stati del processo

```
             Crea()
               │
               ▼
           [PRONTO] ◄──────────── Notifica()
               │                      ▲
    Avvia()    │                      │
               ▼                      │
         [ESECUZIONE] ──Attendi()──► [ATTESA]
               │
    Prerilascio│→ [PRONTO]
               │
           Termina()
               │
               ▼
         [TERMINATO]
```

| Transizione | Descrizione |
|---|---|
| **Crea()** | Inserisce il processo nella ready list in stato PRONTO |
| **Avvia()** | Dispatcher assegna la CPU → ESECUZIONE |
| **Prerilascio** | Quanto esaurito o processo a priorità più alta → PRONTO |
| **Attendi()** | Risorsa non disponibile (I/O, semaforo) → ATTESA |
| **Notifica()** | Risorsa liberata → PRONTO |
| **Termina()** | Il processo rilascia la macchina virtuale |

### fork()

```c
pid_t pid = fork();
// Padre: pid = PID del figlio (> 0)
// Figlio: pid = 0
// Errore: pid = -1
```

- Crea un **clone** del processo con stessa memoria e file aperti
- La memoria diventa **indipendente** dopo la prima modifica (**copy-on-write**)
- Dopo `fork()` il figlio esegue `execve()` per sostituire il codice

**Quanti processi crea N fork() in sequenza senza rami?** $2^N$ processi totali.

**Esempio con rami:**
```c
if (fork() == 0) { fork(); }  // Processi: padre, figlio1, figlio1's-figlio = 3
```

### Thread

Un **thread** è un flusso di controllo indipendente all'interno di un processo.

**Condivisi tra thread dello stesso processo:**
- Spazio di indirizzamento (codice, dati globali, heap)
- File aperti, variabili globali

**Privato per ogni thread:**
- Stack (record di attivazione)
- Registri CPU (contesto)
- Stato (pronto / in esecuzione / in attesa)

| | Processo | Thread |
|---|---|---|
| Spazio di indirizzamento | Privato | Condiviso |
| Creazione | Costosa (fork) | Economica |
| Context switch | Pesante | Leggero |
| Comunicazione | IPC | Variabili condivise |

### Thread POSIX (pthread)

```c
pthread_create(&tid, NULL, funzione, argomento);  // crea thread
pthread_join(tid, NULL);                           // attende terminazione
pthread_exit(NULL);                                // thread termina sé stesso
pthread_mutex_lock(&m);                            // P (acquisisce mutex)
pthread_mutex_unlock(&m);                          // V (rilascia mutex)
```

## 3. Scheduling

### Componenti

| Componente | Ruolo |
|---|---|
| **Scheduler** | Decide quale processo eseguire |
| **Dispatcher** | Attua la scelta: salva contesto uscente, installa entrante (context switch) |

### Metriche

| Metrica | Definizione |
|---|---|
| **Tempo di arrivo** | Istante in cui il processo entra nel sistema |
| **Tempo di attesa** | Permanenza totale in stato PRONTO |
| **Tempo di turn-around** | Completamento − Arrivo (dalla creazione alla terminazione) |
| **Tempo di risposta** | Prima risposta − Arrivo (importante per sistemi interattivi) |
| **Throughput** | Processi completati per unità di tempo |

> Tempo di turn-around = Tempo di attesa + Tempo di esecuzione (+ eventuali prerilasci)

### Algoritmi per sistemi batch

#### FCFS — First Come First Served

- **Senza prerilascio**, senza priorità
- Ordine di esecuzione = ordine di arrivo
- Penalizza i processi I/O-bound se arrivano dopo un processo CPU-bound lungo

**Esempio:** P1(arrivo 0, durata 24), P2(arrivo 0, durata 3), P3(arrivo 0, durata 3)
```
Gantt: [P1: 0–24][P2: 24–27][P3: 27–30]
Attesa: P1=0, P2=24, P3=27  →  Media = 17
```

#### SJF — Shortest Job First

- **Senza prerilascio**; esegue prima il job con durata rimanente minore al momento della scelta
- Ottimizza il throughput e riduce il tempo di attesa medio
- Non equo con lavori lunghi arrivati tardi

#### SRTN — Shortest Remaining Time Next

- **Con prerilascio**: all'arrivo di un nuovo processo si confronta il suo tempo residuo con quello del processo in esecuzione
- Se il nuovo processo è più breve → prerilascio immediato

### Algoritmi per sistemi interattivi

#### Round Robin (RR)

- **Con prerilascio**, senza priorità; ogni processo esegue al più per un **quanto q**
- Coda circolare; processo che esaurisce il quanto torna in fondo alla coda
- Se il processo termina o si blocca prima del quanto, libera volontariamente la CPU

**Regola pratica:** quanto troppo grande → degenera in FCFS; quanto troppo piccolo → overhead eccessivo di context switch.

#### Round Robin con priorità (OQP)

- Code separate per livello di priorità (es. 0 = più alta, 31 = più bassa)
- All'interno della stessa coda: Round Robin
- Un processo a priorità più alta preleva la CPU da uno a priorità più bassa
- Priorità possono essere **statiche** (assegnate alla creazione) o **dinamiche** (cambiano durante l'esecuzione)

### Come costruire il diagramma di Gantt

1. Disegna l'asse del tempo da 0 a T
2. Ad ogni istante scegli il processo da eseguire secondo l'algoritmo
3. Segna quando un processo arriva, viene eseguito, viene prerilasciato o termina
4. Calcola le metriche sommando i tempi di attesa e turn-around

**Esempio RR con q=2:**

Processi: P1(arr=0, dur=5), P2(arr=0, dur=3), P3(arr=0, dur=4)
```
Gantt: [P1:0-2][P2:2-4][P3:4-6][P1:6-8][P2:8-9][P3:9-11][P1:11-12]
```

### Inversione di priorità

**Scenario:** tre processi L (bassa), M (media), H (alta priorità); L e H condividono una risorsa R.

1. L acquisisce R
2. H diventa attivo → vuole R → **H deve attendere L**
3. M diventa attivo → preleva la CPU a L (M ha priorità su L)
4. H deve attendere che M finisca, poi che L rilasci R → **starvation di H**

**Caso reale:** Mars Pathfinder (1997) — frequenti reset di sistema causati da inversione di priorità in VxWorks.

**Soluzione 1 — BPI (Basic Priority Inheritance):**
- Quando H si blocca sulla risorsa di L, L eredita temporaneamente la priorità di H
- L completa la sezione critica senza essere interrotto da M
- BPI **non previene il deadlock**

**Soluzione 2 — ICP (Immediate Ceiling Priority):**
- Ogni risorsa ha un **priority ceiling** = massima priorità tra i processi che possono usarla
- Un processo acquisisce una risorsa solo se la sua priorità è superiore al ceiling di tutte le risorse già in possesso di altri
- Un processo ad alta priorità può essere bloccato **al più una volta**
- ICP **previene il deadlock**

## 4. Sincronizzazione tra Processi

### Il problema della race condition

Quando più processi condividono risorse o variabili, il risultato dipende dalla temporizzazione relativa delle operazioni → **race condition**.

**Sezione critica:** porzione di codice che accede a una risorsa condivisa — deve essere eseguita in **mutua esclusione**.

### Condizioni di ammissibilità

Una soluzione è ammissibile se e solo se:
1. **Accesso esclusivo** — al più un processo alla volta nella sezione critica
2. **Attesa finita** — un processo in attesa entra nella sezione critica in tempo finito
3. **Nessuna assunzione sull'ambiente** — non dipende dalla velocità della CPU o dal numero di core
4. **Indipendenza dall'esterno** — i processi fuori dalla sezione critica non bloccano gli altri

### Semafori (Dijkstra, 1965)

Variabile intera con due operazioni **atomiche**:

| Operazione | Descrizione |
|---|---|
| **P (down, wait)** | Decrementa il contatore; se < 0 → sospende il processo |
| **V (up, signal)** | Incrementa il contatore; se ≤ 0 → sveglia il primo processo in coda |

**Semaforo binario (mutex):** inizializzato a 1; garantisce accesso esclusivo.

**Semaforo contatore:** inizializzato a N; conta le unità disponibili di una risorsa.

```c
void P(sem s) {          void V(sem s) {
    s.valore--;               s.valore++;
    if (s.valore < 0)         if (s.valore <= 0)
        suspend(self);            ready(get(s.coda));
}                         }
```

**Interpretazione del valore:**
- `valore > 0`: N unità ancora disponibili
- `valore = 0`: risorsa esaurita, nessuno in attesa
- `valore < 0`: |valore| processi in coda d'attesa

### Perché sleep/wakeup non bastano

`sleep()` e `wakeup()` possono perdere segnali: se la `wakeup` arriva prima della `sleep`, viene persa → possibile blocco permanente. I semafori risolvono questo con il contatore atomico.

### Produttore-Consumatore con semafori

```c
semaforo mutex     = 1;  // mutua esclusione sul buffer
semaforo non_pieno = N;  // slot liberi nel buffer
semaforo non_vuoto = 0;  // elementi presenti nel buffer

void produttore() {
    while (1) {
        prod = produci();
        P(&non_pieno);   // attende uno slot libero
        P(&mutex);
        inserisci(prod);
        V(&mutex);
        V(&non_vuoto);   // segnala elemento disponibile
    }
}
void consumatore() {
    while (1) {
        P(&non_vuoto);   // attende un elemento
        P(&mutex);
        prod = preleva();
        V(&mutex);
        V(&non_pieno);   // segnala slot liberato
        consuma(prod);
    }
}
```

> **CRITICO:** l'ordine delle P non può essere invertito. Se il produttore fa `P(&mutex)` prima di `P(&non_pieno)`, e il buffer è pieno, si blocca tenendo il mutex → deadlock.

### Lettori e Scrittori

```c
semaforo db    = 1;  // accesso esclusivo ai dati
semaforo mutex = 1;  // protezione del contatore rc
int rc = 0;          // numero di lettori attivi

void reader() {
    P(&mutex); rc++;
    if (rc == 1) P(&db);   // primo lettore esclude scrittori
    V(&mutex);
    /* leggi */
    P(&mutex); rc--;
    if (rc == 0) V(&db);   // ultimo lettore libera scrittori
    V(&mutex);
}
void writer() {
    P(&db); /* scrivi */ V(&db);
}
```

**Problema:** possibile **starvation degli scrittori** se lettori arrivano continuamente.

### Filosofi a cena

N filosofi a tavolo circolare; ogni filosofo ha 1 forchetta alla destra; servono **2 forchette** per mangiare.

| Soluzione | Problema |
|---|---|
| Prendere prima destra poi sinistra | **Deadlock** se tutti prendono la destra contemporaneamente |
| Rilasciare se la seconda non è disponibile | **Starvation** possibile |
| **Mutex globale** attorno all'acquisizione | Corretto ma riduce parallelismo |
| **Filosofo mancino** (uno prende prima la sinistra) | Corretto, parallelismo migliore |

```c
// Soluzione con mutex globale
semaforo mutex = 1;
semaforo forchetta[N] = {1, 1, ..., 1};

void filosofo(int i) {
    while (1) {
        pensa();
        P(&mutex);
        P(&forchetta[i]);
        P(&forchetta[(i+1) % N]);
        V(&mutex);
        mangia();
        V(&forchetta[i]);
        V(&forchetta[(i+1) % N]);
    }
}
```

### Come tracciare l'esecuzione di codice con semafori

**Metodo sistematico:**
1. Disegna una colonna per ogni processo; una riga per ogni istante/passo
2. Per ogni `P(s)`: decrementa s; se s < 0 → il processo si blocca
3. Per ogni `V(s)`: incrementa s; se s ≤ 0 → sveglia il primo in coda
4. Marca il processo sbloccato come pronto; eseguilo al prossimo passo disponibile

### Monitor (Hoare, 1974)

Struttura linguistica che garantisce automaticamente la mutua esclusione:
- Solo un processo alla volta è attivo nel monitor
- Le variabili interne sono accessibili solo ai sottoprogrammi del monitor

**Variabili di condizione:**

| Operazione | Comportamento |
|---|---|
| `wait(cond)` | Sospende il chiamante e rilascia il monitor |
| `signal(cond)` | Risveglia il primo processo in attesa su quella condizione |

`signal` non ha memoria: va perso se nessuno aspetta (a differenza di V sui semafori).

```pascal
monitor PC
    condition non_vuoto, non_pieno;
    integer contenuto := 0;

    procedure inserisci(prod);
    begin
        if contenuto = N then wait(non_pieno);
        <inserisci>; contenuto++;
        if contenuto = 1 then signal(non_vuoto);
    end;

    function preleva : integer;
    begin
        if contenuto = 0 then wait(non_vuoto);
        preleva := <preleva>; contenuto--;
        if contenuto = N-1 then signal(non_pieno);
    end;
end monitor;
```

## 5. Deadlock (Stallo)

### Definizione

Un insieme di processi è in **deadlock** quando ogni processo dell'insieme è in attesa di un evento che solo un altro processo dell'insieme può generare, e nessuno può procedere.

### 4 Condizioni di Coffman (1971) — necessarie e sufficienti

Il deadlock si verifica se e **solo se** tutte e 4 le condizioni sono soddisfatte simultaneamente:

| # | Condizione | Descrizione |
|---|---|---|
| 1 | **Accesso esclusivo** | Almeno una risorsa è detenuta in modalità non condivisibile |
| 2 | **Accumulo di risorse** | I processi possono acquisire nuove risorse senza rilasciare quelle già in possesso |
| 3 | **Inibizione di prerilascio** | Le risorse non possono essere tolte forzatamente; vengono rilasciate solo volontariamente |
| 4 | **Attesa circolare** | Esiste una catena chiusa P₁→R₁→P₂→R₂→...→Pₙ→Rₙ→P₁ |

> Negare **una sola** condizione è sufficiente per impedire il deadlock.

### Grafo di assegnazione delle risorse

**Nodi e archi:**
- **Cerchio** = processo (P₁, P₂, ...)
- **Rettangolo con pallini** = risorsa (ogni pallino = un'istanza)
- **Arco P→R** = il processo richiede la risorsa
- **Arco R→P** = la risorsa è assegnata al processo

**Regole di rilevazione:**
- Se il grafo ha un **ciclo** e tutte le risorse nel ciclo hanno **molteplicità 1** → **deadlock certo**
- Se le risorse hanno molteplicità > 1: un ciclo non implica necessariamente deadlock (serve analisi approfondita)
- Se il grafo **non ha cicli** → nessun deadlock

**Esempio di deadlock (risorse con molteplicità 1):**
```
R1 ──assegnata──► P2 ──richiede──► R2
 ▲                                  │
 │                              assegnata
 └──richiede──── P1 ◄──────────────┘
```
Ciclo P1→R2→P2→R1→P1 → stallo.

**Esempio senza deadlock:** se R1 ha 2 istanze e la seconda è libera, P1 può ottenere R1 senza aspettare → il ciclo si spezza.

### Strategie per affrontare il deadlock

| Strategia | Descrizione | Usato in |
|---|---|---|
| **Prevenzione** | Impedire almeno una delle 4 condizioni strutturalmente | Sistemi batch/semplici |
| **Evitamento (Banchiere)** | Prima di ogni assegnazione verifica che lo stato rimanga sicuro | Sistemi specializzati |
| **Rilevazione e recupero** | Permettere il deadlock, rilevarlo periodicamente, recuperare | Sistemi general-purpose |
| **Indifferenza** | Ignorare il problema | Maggioranza degli S/O moderni |

### Algoritmo del Banchiere (Dijkstra) — FONDAMENTALE

#### Strutture dati

`N` = numero processi, `M` = numero tipi di risorse

| Struttura | Dimensione | Significato |
|---|---|---|
| **Disponibili** | M | `Disponibili[j]` = istanze libere di Rⱼ |
| **Massimo** | N × M | `Massimo[i,j]` = max istanze di Rⱼ che Pᵢ potrà mai richiedere |
| **Assegnate** | N × M | `Assegnate[i,j]` = istanze di Rⱼ attualmente assegnate a Pᵢ |
| **Necessità** | N × M | `Necessità[i,j] = Massimo[i,j] − Assegnate[i,j]` |

#### Algoritmo di verifica della sicurezza

```
1. Lavoro[j] = Disponibili[j]  per ogni j
   Fine[i]   = false           per ogni i

2. Cerca i tale che:
   (a) Fine[i] == false
   (b) Necessità[i,j] ≤ Lavoro[j]  per ogni j
   Se non esiste i → vai a 4.

3. Lavoro[j] += Assegnate[i,j]   (il processo i termina e rilascia)
   Fine[i] = true
   Torna a 2.

4. Se Fine[i] == true per ogni i → STATO SICURO
   Altrimenti → STATO NON SICURO (possibile deadlock)
```

**Complessità:** O(M · N²)

#### Algoritmo di richiesta delle risorse

Quando Pᵢ richiede `Richieste[i]`:

```
I.   Se Richieste[i] > Necessità[i]  → ERRORE (richiesta oltre il massimo dichiarato)

II.  Se Richieste[i] > Disponibili   → Pᵢ deve attendere (risorse non disponibili)

III. Simula l'assegnazione:
     Disponibili  -= Richieste[i]
     Assegnate[i] += Richieste[i]
     Necessità[i] -= Richieste[i]

IV.  Esegui l'algoritmo di sicurezza:
     Se SICURO  → assegna definitivamente le risorse
     Se INSICURO → Pᵢ attende; annulla la simulazione (ripristina i valori)
```

#### Esempio completo

**Sistema:** 4 processi (A, B, C, D), 5 tipi di risorse.
`Disponibili = [0, 1, 3, 1, 2]`

| Processo | Assegnate | Massimo | Necessità |
|---|---|---|---|
| A | 1 0 2 1 1 | 3 1 2 1 3 | **2 1 0 0 2** |
| B | 2 0 1 1 1 | 3 3 4 2 1 | **1 3 3 1 0** |
| C | 1 1 0 1 0 | 2 1 4 1 0 | **1 0 4 0 0** |
| D | 1 1 1 1 0 | 1 1 3 2 1 | **0 0 2 1 1** |

**Esecuzione dell'algoritmo** (Lavoro iniziale = `[0 1 3 1 2]`):

| Passo | Processo | Necessità ≤ Lavoro? | Nuovo Lavoro |
|---|---|---|---|
| 1 | **D** | `[0 0 2 1 1]` ≤ `[0 1 3 1 2]` ✓ | `[0+1, 1+1, 3+1, 1+1, 2+0]` = `[1 2 4 2 2]` |
| 2 | **C** | `[1 0 4 0 0]` ≤ `[1 2 4 2 2]` ✓ | `[1+1, 2+1, 4+0, 2+1, 2+0]` = `[2 3 4 3 2]` |
| 3 | **A** | `[2 1 0 0 2]` ≤ `[2 3 4 3 2]` ✓ | `[2+1, 3+0, 4+2, 3+1, 2+1]` = `[3 3 6 4 3]` |
| 4 | **B** | `[1 3 3 1 0]` ≤ `[3 3 6 4 3]` ✓ | — |

**Sequenza sicura: D → C → A → B** → stato sicuro.

#### Stato sicuro vs stato non sicuro

- **Stato sicuro:** esiste almeno una sequenza di esecuzione in cui tutti i processi completano
- **Stato non sicuro:** non è possibile garantire l'assenza di deadlock (ma non implica deadlock immediato)
- **Deadlock** ⊂ **stati non sicuri** ⊂ **tutti gli stati**

## 6. Domande Vero/Falso Frequenti agli Esami

| Affermazione | Risposta |
|---|---|
| Un ciclo nel grafo di assegnazione implica sempre deadlock | **FALSO** — solo se le risorse hanno molteplicità 1 |
| Uno stato non sicuro implica necessariamente deadlock | **FALSO** — solo che non si può garantire l'assenza |
| BPI previene il deadlock | **FALSO** — previene solo l'inversione di priorità (non il deadlock) |
| ICP previene il deadlock | **VERO** |
| Il processo con più risorse allocate va terminato per primo per risolvere un deadlock | **FALSO** — si sceglie il processo "meno costoso" da terminare (vari criteri) |
| SRTN e SJF producono lo stesso risultato in assenza di arrivi dopo t=0 | **VERO** — senza nuovi arrivi il prerilascio non avviene mai |
| Round Robin con quanto infinito equivale a FCFS | **VERO** |
| Il valore negativo di un semaforo indica il numero di processi in attesa | **VERO** — `|valore|` processi in coda |
| `signal` su una variabile di condizione del monitor va perso se nessuno aspetta | **VERO** — a differenza di V su semaforo |
| I thread di uno stesso processo condividono la heap | **VERO** — condividono heap, codice e file; ogni thread ha il suo stack |
| `fork()` restituisce 0 al padre e il PID del figlio al figlio | **FALSO** — è il contrario: 0 al figlio, PID al padre |

## 7. Riepilogo Metriche di Scheduling

**Formule:**

$$\text{Attesa}_i = \text{Inizio esecuzione}_i - \text{Arrivo}_i \quad \text{(senza prerilascio)}$$

$$\text{Turn-around}_i = \text{Fine}_i - \text{Arrivo}_i$$

$$\text{Attesa media} = \frac{\sum_i \text{Attesa}_i}{N}$$

**Con prerilascio**, il tempo di attesa è la somma di tutti i periodi in stato PRONTO.

## 9. Schemi Rapidi per gli Esercizi

### Algoritmo del Banchiere — procedura in 3 passi

1. Calcola la matrice **Necessità** = Massimo − Assegnate
2. Parte da **Lavoro** = Disponibili
3. Cerca iterativamente il processo i con `Fine[i]=false` e `Necessità[i] ≤ Lavoro`; aggiorna Lavoro con le risorse di i; segna `Fine[i]=true`; ripeti. Se tutti i Fine sono true → sicuro.

### Grafo di assegnazione — come leggere il deadlock

1. Costruisci il grafo con archi R→P (assegnazione) e P→R (richiesta)
2. Cerca cicli
3. Se ci sono cicli e tutte le risorse nel ciclo hanno una sola istanza → deadlock certo
4. Se alcune risorse hanno più istanze → verifica se le istanze extra sono libere o assegnate fuori dal ciclo

### Scheduling con Gantt — procedura

1. Metti tutti i processi in una lista con arrivo e durata
2. All'istante t: scegli il processo da eseguire secondo l'algoritmo
3. Avanza il tempo: di 1 unità (per algoritmi con prerilascio) o fino alla fine del processo (FCFS/SJF)
4. All'arrivo di nuovi processi (SRTN/RR): reinserisci nella coda e rivaluta
5. Al termine, calcola attesa = Σ(periodi in stato PRONTO) e turn-around = Fine − Arrivo
